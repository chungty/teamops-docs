Is this a fit?

✓ Good fit

  • Founder-led or small operator-led SaaS
  • One workflow keeps eating attention every week
  • Pain is concrete: routing, handoffs, follow-up, approvals, reporting, support, onboarding, or billing exceptions
  • You want leverage faster than a full ops hire
  • A decision-maker can stay available during the sprint

✕ Not a fit

  • Need a full internal tool before the workflow is even understood
  • The real problem is strategy, not repetitive execution
  • The process changes daily — nothing is stable enough to spec yet
  • No owner available to confirm rules and exceptions
  • You want invented AI magic instead of a real operating document

What "builder-ready" looks like in practice

Illustrative example — not a client case study. This is a truthful excerpt showing the level of specificity the sprint produces.

Sample spec slice · Demo-request routing

Trigger + inputs captured

  • New demo request from website form, founder-forwarded email, or Slack intake
  • Captured: contact, company, role, stated pain, source, timestamp, owner availability
  • Goal: route fast without losing context or creating duplicate follow-up

Decision rules

  • If an open opportunity exists → append context and notify the current owner; do not create a second route
  • If pain is concrete and workflow-shaped → route to automation-spec qualification
  • If the request is vague → send one clarifying question before booking time
  • If the inbound is from an existing customer → route to support/success, not sales

Exceptions + human-in-loop

  • Duplicate submissions inside a 7-day window
  • Missing company or domain → flag for manual review
  • Unclear ownership → escalate rather than silently drop the lead
  • Founder approves final sprint scope before quoting or kickoff

Output state

  • One clear owner with SLA
  • Logged route reason and current state
  • Draft next action for the assigned owner
  • Audit trail that makes later automation implementation straightforward

How the five days work

Day 1
Choose the workflowPick the one live, painful, and worth-fixing-now process. Agree on scope before building anything.
Day 2
Map the current stateWrite down what actually happens today — not the ideal, not the wiki, but the real sequence.
Day 3
Write the specDefine the trigger, actions, rules, exceptions, approvals, and owner handoffs in plain English.
Day 4
Pressure-test the edgesReview failure modes, rollback paths, and where humans still need to stay in the loop.
Day 5
HandoffShip the final spec plus the immediate implementation recommendation — what to build now, what to test first, what stays manual.
